What this shows

The exact same code runs in both environments — only where each process lives is different. Locally you manage the infrastructure yourself; in production Railway hosts it all.

🐳 Docker (local only) — spins up PostgreSQL so you don't need it installed. One docker-compose up -d and the DB is ready.
🚀 Railway (production) — each box in the diagram is a separate Railway service. The Postgres plugin provides a managed DB with an internal connection URL auto-injected as DATABASE_URL.
🔒 Worker has no public URL — it's a background process that only talks outward (to monitored URLs) and inward (to the DB). No HTTP server, no port.
⚡ Railway auto-deploys on every push to main, so shipping is just merging a PR.

Overall System Architecture
flowchart LR Browser["🌐 Browser\nReact + Vite\nTanStack Query"] subgraph API["Express API · Node.js"] MW["Middleware\nhelmet · cors · jwt"] RT["Routes\n/api/auth\n/api/monitors\n/api/users\n/api/users/me/settings"] SVC["Services\nmonitor · alert · email"] REPO["Repositories\nmonitor · check\nincident · alert"] end subgraph Worker["Cron Worker · Node.js"] UW["⏱ Uptime\n1 min"] SW["⏱ SSL\n1 hour"] HW["⏱ Headers\n6 hours"] end DB[(PostgreSQL\nvia Prisma)] EXT["🌍 External URLs"] EMAIL["📧 Nodemailer\nSMTP"] TYPES["📦 shared-types\nZod · TypeScript"] Browser -->|"Bearer JWT"| MW MW --> RT --> SVC --> REPO --> DB UW & SW & HW -->|HTTP / TLS| EXT UW & SW & HW --> REPO SVC --> EMAIL TYPES -.->|shared| Browser TYPES -.->|shared| API

What this shows

The three main processes and how they relate. The API serves the browser; the Worker runs silently in the background; shared types keep both ends of the stack in sync.

🌐 Frontend — pure SPA. It never touches the DB directly; everything goes through the API over HTTP with a JWT in the Authorization header.
🔁 Layered backend — Routes → Services → Repositories. Routes handle HTTP, Services hold business logic, Repositories are the only things that touch Prisma/DB.
⏱ Worker is completely independent — no HTTP server, no shared memory with the API. It reads monitors from the DB and writes check results back. That's it.
📦 shared-types — a local package (@watchdog/shared-types) with Zod schemas and TypeScript types used by both frontend and backend, preventing drift.

Worker Pipeline — What Each Worker Does
flowchart TD TICK1["⏱ every 1 min"] --> UW["Uptime Worker"] TICK2["⏱ every 1 hour"] --> SW["SSL Worker"] TICK3["⏱ every 6 hours"] --> HW["Headers Worker"] UW -->|"GET url · 10s timeout"| URL1["Target URL"] SW -->|"TLS handshake · port 443"| URL2["Target URL"] HW -->|"GET url · read headers"| URL3["Target URL"] URL1 --> CHK1["Check · uptime\nresponseTime · statusCode"] URL2 --> CHK2["Check · ssl\nsslDaysLeft"] URL3 --> CHK3["Check · headers\npresent / missing"] CHK1 -->|"down + no open incident"| INC["Incident created"] CHK2 -->|"daysLeft ≤ 14 + no open incident"| INC CHK1 -->|"back up + open incident"| RES["Incident resolved ✓"] CHK2 -->|"daysLeft > 14 + open incident"| RES INC --> AS["AlertService\ncooldown check"] AS -->|"enabled + not yet sent"| EM["📧 Email sent"] AS --> AR["Alert record saved"]

What this shows

All three workers follow the same pattern: poll every monitor → write a Check record → decide if an Incident should be opened or closed → trigger an alert if needed.

🔴 Uptime worker — HTTP GET every minute. If the response status is ≥ 400 or the request times out, the monitor is "down". Records response time and status code every run.
🔒 SSL worker — opens a raw TLS connection to port 443, reads the certificate's valid_to field, and calculates days remaining. Warns at ≤ 14 days.
🛡️ Headers worker — fetches the URL and inspects the response for security headers: CSP, HSTS, X-Frame-Options, and others. Saves which are present and which are missing.
🔕 Alert cooldown — one email per incident, ever. The Alert record acts as a sent-flag so the worker never spams you even if it keeps detecting the same problem.

API Request Flow (authenticated endpoint)
sequenceDiagram participant B as Browser participant MW as Auth Middleware participant RT as Route Handler participant SVC as Service participant REPO as Repository participant DB as PostgreSQL B->>MW: GET /api/monitors\nAuthorization: Bearer jwt MW->>MW: jwt.verify(token, JWT_SECRET) alt invalid / missing token MW-->>B: 401 Unauthorized end MW->>RT: req.user = { id, email } RT->>SVC: monitorService.listByUser(req.user.id) SVC->>REPO: monitorRepository.findByUser(userId) REPO->>DB: SELECT * FROM monitors\nWHERE userId = ? DB-->>REPO: rows REPO-->>SVC: Monitor[] SVC-->>RT: Monitor[] RT-->>B: 200 { success: true, data: [] }

What this shows

A single authenticated API call traced from the browser all the way to Postgres and back. Every protected route follows this exact chain.

🔑 JWT in every request — after login the frontend stores the token and attaches it as Authorization: Bearer <token> on every API call via an axios interceptor.
🚧 Middleware gates the route — authenticate() verifies the token before the route handler even runs. Invalid → 401, no further code executes.
🪜 Strict layering — the Route handler never calls Prisma directly. It calls a Service, which calls a Repository. This keeps business logic and DB queries separate and testable.
👤 User ID scoping — the Service always filters by req.user.id, so a logged-in user can only ever see their own monitors. There's no admin bypass in the current codebase.

Database Schema (Prisma / PostgreSQL)
erDiagram User { uuid id PK string email string password string name string alertEmail bool alertDowntime bool alertSslExpiry } Monitor { uuid id PK uuid userId FK string name string url int intervalMinutes bool isActive } Check { uuid id PK uuid monitorId FK enum type string status int statusCode int responseTime int sslDaysLeft json headers datetime checkedAt } Incident { uuid id PK uuid monitorId FK enum type datetime startedAt datetime resolvedAt bool isResolved } Alert { uuid id PK uuid userId FK uuid incidentId FK string channel datetime sentAt } User ||--o{ Monitor : "owns" Monitor ||--o{ Check : "produces" Monitor ||--o{ Incident : "triggers" User ||--o{ Alert : "receives" Incident ||--o{ Alert : "generates"

What this shows

Five tables and how they connect. Everything in the system — checks, incidents, alerts — traces back to a Monitor, which traces back to a User.

👤 User — stores credentials and alert preferences (alertDowntime, alertSslExpiry). The optional alertEmail lets you route alerts to a different address than your login email.
📡 Monitor — a URL you want to watch. Has an isActive flag so you can pause monitoring without deleting history.
📋 Check — one row per worker run per monitor. The type enum tells you which worker wrote it: uptime, ssl, or headers. This table grows the fastest.
🚨 Incident — created when something goes wrong, resolved when it recovers. The resolvedAt / isResolved pair tracks the full window of an outage.
📧 Alert — one row per email sent. Acts as both an audit log and a deduplication key — if a row exists for an incident, no second email is sent.
